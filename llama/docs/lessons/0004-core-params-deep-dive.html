核心参数原理详解

今天把前几课见过的参数全部拆开：它们控制推理过程中的哪一个环节？改它们到底会发生什么？

与你的任务的关系：你已经知道 -ngl 控制 GPU offload，-c 控制上下文大小，--temp 控制随机性。但真正调参时，你需要知道：-b 和 -ub 有什么区别？flash-attn 省什么？-np 开多会怎样？这节课让你从"知道参数名"到"理解参数为什么这样设"。

1. 回顾推理流水线（把参数放回去）

第一课的心智模型现在加上参数位置：

GGUF 模型加载 —— 相关参数：-m, -hf, --mmap, --mlock │ 权重放到哪？—— -ngl (GPU offload 层数) 计算在哪做？—— -dev (设备选择), -t (CPU 线程) │ Prompt prefill —— -b (batch), -ub (ubatch), -tb (threads-batch) 并行处理 prompt token，建立 KV cache │ Token decode loop —— -t, -fa, -ctk/-ctv 逐 token 生成，依赖 KV cache │ 采样选 token —— --temp, --top-p, --top-k, --min-p, --repeat-penalty, --mirostat │ Server/API 层 —— --host, --port, -np (并行 slot), --api-key

2. 深度拆解：GPU offload 相关

-ngl, --n-gpu-layers

官方定义："max. number of layers to store in VRAM, either an exact number, 'auto', or 'all' (default: auto)"

  • 原理：Transformer 模型由多层组成。每层包含 attention + FFN 计算。把层放在 GPU 上，矩阵运算走 CUDA 核心，速度远快于 CPU。
  • 不是 0/1 开关：你可以只 offload 前 20 层，留 12 层在 CPU。这称为 CPU+GPU 混合推理。
  • "auto" 含义：新版本默认 auto，让 llama.cpp 根据你显存自动决策。
  • "all" / 大数字：如果设 99 或 all，llama.cpp 会尽量全部 offload；如果显存不够，它会自动减少。
  • 典型场景：5060 Ti 16GB，7B Q5_K_M 约 5.5GB → 全部 offload 通常可行。如果模型更大或上下文更长，降低 -ngl 来节省显存。

-fa, --flash-attn

官方定义："set Flash Attention use ('on', 'off', or 'auto', default: 'auto')"

  • 省什么：Flash Attention 是一种更高效的注意力计算方式，主要节省显存（KV cache 相关），有时也能提升速度。
  • 什么时候重要：上下文越长，Flash Attention 节省效果越明显。8K+ 上下文场景强烈建议开启。
  • auto 模式：如果检测到你的 GPU 支持（计算能力 8.0+），auto 会自动启用。

3. 深度拆解：上下文与 KV cache

-c, --ctx-size

官方定义："size of the prompt context (default: 0, 0 = loaded from model)"

  • 原理：上下文大小 = 模型一次会话能"看到"的最大 token 数。包括系统提示、历史对话、当前输入、模型输出。
  • 为什么不是越大越好：KV cache 大小与上下文成正比。上下文翻倍，KV cache 翻倍。
  • 默认值 0：表示使用模型元数据里标注的上下文长度（例如 8192 或 128K）。
  • 你的 5060 Ti：设 4096 或 8192 通常安全。如果模型原生支持 128K 上下文（如一些新模型），直接开到 128K 会让 KV cache 吃掉几乎所有显存。

-ctk, -ctv — KV cache 量化

官方定义：KV cache data type for K/V (default: f16)，支持 f32, f16, bf16, q8_0, q4_0 等。

  • 原理：KV cache 默认用 f16 存储。如果把 KV cache 量化到 q8_0 或 q4_0，能大幅降低 KV cache 的显存占用，代价是轻微的质量损失。
  • 实用建议：上下文超过 32K 或并发数高时，考虑 -ctk q8_0 -ctv q8_0 甚至 q4_0 来节省显存。

4. 深度拆解：Batch 与并发

-b, --batch-size vs -ub, --ubatch-size

这两个参数初学者最容易混淆。官方定义：

  • -b：logical maximum batch size (default: 2048)
  • -ub：physical maximum batch size (default: 512)

为什么有两个 batch？

Prompt processing（prefill）阶段需要并行计算多个 token 的注意力。但显存有限，不能一次把所有 token 全部并行算。所以：

  • 逻辑 batch（-b）：你希望一次处理多少 token。比如 prompt 有 4000 个 token，设 -b 2048，那需要至少 2 个 step 处理完。
  • 物理 batch（-ub）：每一步实际送入 GPU 的 token 数。如果设 -ub 512，那即使 -b 是 2048，每一步也只送 512 个 token 进 GPU 计算。

直觉：-ub 是显存安全阀。如果 OOM，先降 -ub。如果 prompt 处理慢，可以尝试提高 -b（但效果取决于模型和 GPU）。

-np, --parallel

官方定义："number of server slots (default: -1, -1 = auto)"

  • 原理：每个 slot 对应一个并发请求。如果 -np 4，最多 4 个请求同时处理。
  • 代价：每个 slot 都有独立的 KV cache。如果 -c 8192、-np 4，KV cache 总需求 ≈ 4 × 8K 的 KV cache。你的 16GB 显存需要精打细算。
  • auto 模式：新版本会根据显存自动计算 slot 数。
  • continuous batching：当 -cb 启用（默认），llama.cpp 会动态在 batch 中混合多个请求的 token，提高 GPU 利用率。

5. 深度拆解：采样参数（控制输出风格）

官方 server README 把采样参数列在独立章节，因为它们在推理计算之后、返回 token 之前起作用。

--temp, --temperature

默认：0.80

对 logits 做 softmax 前，除以 temperature。温度越低（如 0.1），高概率 token 的胜出优势越大；温度越高（如 1.5），各 token 概率更均匀，输出更随机。

Agent 场景：通常 0.6–0.8 之间。需要确定性输出时可降到 0.1–0.3。

--top-p

默认：0.95

把所有 token 按概率从高到低排序，从累计概率达到 p 的最小集合中采样。p=0.95 意味着忽略最不靠谱的 5% token。

和 top-k 的区别：top-k 固定选 k 个；top-p 动态选，分布尖锐时选得少，分布平坦时选得多。

--top-k

默认：40

只保留概率最高的 k 个 token，把其他的概率置零。k=40 是经典值；k=1 等于 greedy decoding（总是选最可能那个）。

--min-p

默认：0.05

忽略概率低于（最高概率 × min-p）的 token。例如最高概率 token 是 0.5，min-p=0.05 则忽略概率低于 0.025 的所有 token。

对比：相比 top-p，min-p 对低质量模型更友好——不会在概率平坦时误砍可用 token。

采样顺序很重要

官方参数 --samplers 默认值为：penalties;dry;top_n_sigma;top_k;typ_p;top_p;min_p;xtc;temperature

这意味着采样器按此顺序依次处理 logits。例如 --top-p 在 --top-k 之后应用，先截断到 top-k 个 token，再在剩下的里面做 top-p 过滤。

6. 综合场景：一个 5060 Ti 16GB 的推荐启动命令

llama-server ^
  -m C:\models\my-8b-instruct.Q5_K_M.gguf ^
  --host 127.0.0.1 --port 8080 ^
  -ngl all ^
  -c 8192 ^
  -fa on ^
  -b 2048 -ub 512 ^
  -np 1 ^
  -t 8 ^
  --temp 0.7 --top-p 0.9

解释每项选择：

  • -ngl all：16GB 显存，8B Q5_K_M ≈ 5.5GB，全部 offload 可行。
  • -c 8192：8K 上下文，KV cache 约 4GB，仍在余量内。
  • -fa on：明确开启 Flash Attention，节省 KV cache 显存。
  • -np 1：单并发。先保证链路稳定；后续再根据 Agent 需求调整。
  • -t 8：8 个 CPU 线程。即使大部分在 GPU 上，采样等仍需要 CPU。
  • --temp 0.7 --top-p 0.9：温和随机性，适合 Agent 对话。
