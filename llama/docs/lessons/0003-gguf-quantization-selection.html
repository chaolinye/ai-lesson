如何选择 GGUF 模型与量化版本

今天的胜利条件：你能为一个 5060 Ti 16GB 环境推荐一套“模型 + 量化”组合，并解释为什么不是别的组合。

与你的任务的关系：你上一课已经能把 server 跑起来。但如果模型选错或量化选错，要么显存爆炸，要么输出质量下降。这节课把“选模型”从玄学变成决策。

1. 量化到底是什么？——不只是“压缩包”

量化不是 zip 那种无损压缩。官方 quantize README 明确说：量化通过降低权重精度来缩小体积，但可能引入 accuracy loss。官方常用 perplexity (ppl) 和 Kullback–Leibler Divergence (kld) 来衡量这种损失。

不是“精度变低就会胡说”
量化主要影响模型权重的表示精度。对很多任务来说，Q5_K_M 与 FP16 的差异人类很难感知。
但量化不是免费的
更低 bit 数意味着更大的信息损失。Q4_K_M 比 Q5_K_M 省约 0.7GB (7B 模型)，但 perplexity 差距也更大。

2. 读懂量化名称：Q5_K_M 是什么意思？

官方 wiki 和社区共识的解读：

部分含义
Q5每个权重约 5 bit。bit 越高，理论上保留信息越多。
KK-quant，改进版量化方法，比旧版更精细（旧版如 Q4_0、Q4_1 通常不推荐用于新模型）。
M/S/L混合精度档位。M = Medium（常用），S = Small（更省），L = Large（更保真）。并非所有量化都有这三个档位。

旧版量化：Q4_0、Q4_1、Q5_0、Q5_1 是较老的格式。社区对新模型通常优先 K-quants。

3. 官方数据：量化质量与体积对比

基于社区整理的近似数据（以 7B 模型为例，来自官方工具与社区 benchmark）：

量化约大小相对 FP16 ppl 变化一句话定位
FP1613.0 GB—理论基准，通常不直接跑。
Q8_07.0 GB+0.03%近乎无损，小模型或质量敏感场景选它。
Q6_K5.5 GB+0.13%最佳质量/体积折中之一。
Q5_K_M4.8 GB+0.39%16GB 显存首选候选之一，质量与体积兼顾。
Q4_K_M4.1 GB+1.68%社区常用默认推荐，显存紧张或模型更大时的好选择。
Q4_K_S3.9 GB+2.62%极限省空间，但对推理质量影响更明显。
官方 perplexity 工具的关键警告：官方 perplexity README 指出：perplexity 不能跨不同 tokenizer 的模型直接比较。它最适合做同模型不同量化的质量对比。如果你看到网上有人拿 Qwen 的 ppl 和 Llama 的 ppl 比，那个数字本身没有直接意义。

4. 5060 Ti 16GB 的选择策略

我们回到第一课的心智模型：

总 VRAM 占用 ≈ 量化后权重 + KV cache + 计算缓冲 + server/驱动开销

这意味着：

  1. 选 Q5_K_M 还是 Q4_K_M，不只是看“模型文件多大”，还要看你要留多少显存给 KV cache 和并发。
  2. 上下文长度（-c）越长，KV cache 越大，留给权重的预算就越紧。
  3. 并发（-np）越高，KV cache 也成倍增长。
模型大小推荐量化理由（16GB 视角）
7B/8BQ5_K_M 或 Q4_K_M两者都舒适。Q5_K_M 更保真；Q4_K_M 可为长上下文留更多余量。
14BQ5_K_M 或 Q4_K_MQ5_K_M ≈ 9.6GB；Q4_K_M ≈ 8.2GB。若上下文大或并发高，优先 Q4_K_M。
22B/27BQ4_K_M（或更小）Q4_K_M 约 12–14GB，已接近上限。上下文和并发要保守。
32B+Q4_K_M 或更小，或 CPU 混跑即使 Q4_K_M 也可能超过显存；需要仔细计算或降低 -ngl。

5. 模型在哪里找？

Hugging Face Hub 是目前 GGUF 模型最主要的来源。官方 README 说明你可以用 -hf 参数直接从 Hugging Face 拉模型：

llama-server -hf user/model:Q5_K_M

搜索技巧：

  • 在 huggingface.co/models?apps=llama.cpp 筛选 llama.cpp 兼容模型。
  • 优先选 Instruct 后缀的模型，说明它经过微调，更适合对话/chat completion。
  • 看模型卡（Model Card）：确认上下文长度、chat template、license、推荐的 GGUF 版本。
  • 确认模型卡里有 chat template。没有 chat template 的模型，llama-server 可能无法正确把 messages 转成 prompt。

6. 一个决策示例

场景：你有一块 5060 Ti 16GB，想让 Agent 调本地 8B instruct 模型，上下文最多 8K，单并发。

计算：

  • Q5_K_M 权重 ≈ 5.5 GB (8B 模型)
  • KV cache 约 2–4 GB（取决于 8K 上下文、层数、隐藏维度）
  • 计算缓冲 + server 开销 ≈ 1–2 GB
  • 总需求 ≈ 9–11 GB，16GB 显存有余量。

结论：选 Q5_K_M 合理。如果未来并发增加，再评估是否降到 Q4_K_M 或减少上下文。
