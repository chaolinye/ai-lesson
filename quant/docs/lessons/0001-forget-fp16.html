Lesson 1: Forget FP16

为什么需要量化？一个 7B 模型的生存指南

假设你刚下载了一个 7B 模型（比如 Llama-3-7B）。7B = 70 亿个参数。每个参数以 FP16（16 位浮点数）存储。

算一笔账：
7,000,000,000 × 2 bytes = 14 GB — 这只是权重，还没算 KV Cache、中间激活值。
一张 RTX 4090 有 24 GB VRAM，勉强够；一张 RTX 3060 有 12 GB，直接出局。

而如果你把每个参数从 16 bits 压缩到 4 bits：

14 GB → 3.5 GB

现在同样的 7B 模型，一个 8 GB 显存的显卡就能轻松跑起来。这就是量化存在的理由。

量化的核心直觉

量化的本质可以用一句话概括：

用更少的比特数来表示一个数值，接受一定的精度损失，换取内存和速度的巨大收益。

想象你有一把尺子：FP16 的尺子有 65,536 个刻度（2^16），可以非常精细地测量。INT4 的尺子只有 16 个刻度（2^4），你不得不把测量值"四舍五入"到最近的刻度上。

💡 关键 insight：神经网络对权重的微小误差有很高的容忍度。模型不是 MySQL 数据库——它不需要"精确的"权重值，只需要"足够好的"权重值。量化就是利用这个特性，把不必要的精度砍掉。

量化的数学：映射 x → q

量化的核心操作就是一个线性映射（以对称量化为例）：

q = round(x / scale)

其中：

  • x：原始浮点数（FP16）
  • q：量化后的整数（INT4）
  • scale：缩放因子 = max(|x|) / (2bits-1 - 1)

反过来，推理时做反量化：

x̂ = q × scale

x̂ 是重建的浮点数。因为 round() 操作，x̂ ≠ x —— 这之间的差就是量化误差。

一个例子

假设我们有一个权重矩阵的某一行：

x = [0.12, -0.87, 0.53, -0.04, 0.99, -0.33]

# 对称 INT4 量化（范围 -7 ~ 7）
scale = max(|x|) / 7 = 0.99 / 7 ≈ 0.1414

# 量化
q = round(x / scale) = [1, -6, 4, 0, 7, -2]

# 反量化重建
x̂ = q × scale = [0.14, -0.85, 0.57, 0, 0.99, -0.28]

# 误差
error = |x − x̂| = [0.02, 0.02, 0.04, 0.04, 0, 0.05]

最大值 0.99 被精确保存（因为它决定了 scale），但像 -0.04 这样的小值被直接舍入成 0。平均误差约 0.03——对于模型推理来说，这是可以接受的。

动手环节：用 PyTorch 实现基础量化

📝 练习：手写一个对称量化器

打开你的 Python 环境，运行下面的代码。目标是：

  1. 生成一个随机权重矩阵（模拟 LLM 的线性层）
  2. 实现对称 INT4 量化
  3. 观察量化误差的分布
import torch
import numpy as np

def symmetric_quantize(x, bits=4):
    """对称量化：x (FP16) → q (INT4)"""
    # 找最大值确定 scale
    max_val = x.abs().max()
    q_max = 2**(bits - 1) - 1  # INT4 对称范围: -7 ~ 7
    scale = max_val / q_max

    # 量化
    q = torch.round(x / scale).clamp(-q_max, q_max)
    return q, scale

def dequantize(q, scale):
    """反量化：q (INT4) → x̂ (FP16)"""
    return q * scale

# 模拟一个 Llama 7B 的 q_proj 权重（一小块）
torch.manual_seed(42)
weight = torch.randn(64, 64) * 0.1  # 模拟一个 64×64 的权重块

# 量化
q, scale = symmetric_quantize(weight)
weight_recon = dequantize(q, scale)

# 评估误差
mse = ((weight - weight_recon) ** 2).mean()
max_err = (weight - weight_recon).abs().max()
print(f"MSE:  {mse:.6f}")
print(f"Max Error: {max_err:.6f}")
print(f"Scale: {scale:.6f}")
print(f"原始大小: {weight.element_size() * weight.numel() / 1024:.1f} KB")
print(f"量化后:  {q.element_size() * q.numel() / 1024:.1f} KB (但实际存储在 INT4 阵列中)")
试试这个扩展：把 bits 改成 8 看看误差变化。再试试改成 2。观察 MSE 随比特数的变化曲线——为什么 4-bit 是"最佳平衡点"？

量化不是免费的午餐

量化当然有代价。但关键在于——你的目标是什么？

  • 做推理（只是用模型）：4-bit 量化在大多数任务上只损失不到 1% 的性能。值得。
  • 做微调：4-bit + LoRA (QLoRA) 是标准方案。基础权重保持量化，只训练小的适配器。
  • 做数学推理（需要精确计算）：可能需要 8-bit 甚至保持 FP16。
⚠️ 常见坑：很多人第一次量化后发现模型"变蠢了"。原因通常不是量化本身，而是：
① 量化方案选错了（比如用 Q2_K 跑数学题）
② 校准数据集和你的使用场景不匹配
③ 模型本来就很敏感（小模型量化损失更大）
