核心参数原理详解

今天的胜利条件：你能解释每个常用参数“控制推理流程的哪个环节”，而不是只背默认值。

与你的任务的关系：你的 Agent 会调用本地模型。理解参数后，你能根据 Agent 的 prompt 长度、并发量、输出风格需求，有针对性地调整 server 配置，而不是复制粘贴命令。

1. 把参数放回推理流程

第一课我们画过这个流程。现在把参数标上去：

GGUF 模型文件
  │
  ▼
加载权重到内存/显存  ←── 由 -m 指定文件，-ngl 决定放 GPU 的层数
  │
  ▼
Prompt prefill       ←── -b (逻辑 batch) 和 -ub (物理 batch) 控制处理效率
  │     │
  │     └── 建立 KV cache  ←── -c (ctx-size) 决定 KV cache 最大容量
  │
  ▼
Token decode loop    ←── 每步生成一个 token
  │     │
  │     └── logits → 采样参数链 → 选中 token
  │           temp, top-p, top-k, min-p, penalties...
  │
  ▼
输出到 API 响应

下面按环节逐个拆参数。

2. 模型加载：-m 与 -ngl

-m, --model

指定 GGUF 文件路径。建议是绝对路径，避免工作目录变化导致找不到模型。官方 server README 和 cli README 都把它列为首要参数。

-ngl, --n-gpu-layers

官方 multi-gpu 文档说明：这是 "Maximum number of layers to keep in VRAM"，支持 auto、all 或具体数字。默认值在新版本是 auto。

设 all / 999
让 llama.cpp 尽量把层放进 GPU。显存够时是正确选择。
设具体数字
例如 -ngl 20。当你知道总层数且想精确控制显存时。
设 auto
让 llama.cpp 自己估算。通常合理，但若你的显存很紧，建议手动验证。
降低 -ngl
OOM 时的急救方案。部分层回 CPU，速度下降但通常能跑。

原理：模型通常由几十层 Transformer block 组成。-ngl 控制多少层放在 GPU VRAM。如果设 0，所有计算在 CPU 上。如果设总层数，尽量全放 GPU。中间值是混合模式：前 n 层在 GPU，后面在 CPU。

3. 上下文与 KV cache：-c

-c, --ctx-size

官方 server README 定义："size of the prompt context"，默认 0（0 = 从模型元数据加载）。

这个参数是第一课心智模型的核心：它直接决定 KV cache 的最大容量。KV cache 不是模型文件的一部分，是运行时分配的内存/显存。

KV cache 大小的粗略直觉（非精确公式）：
KV cache 大小 ∝ 层数 × 头数 × 每头维度 × 上下文长度 × 并发数 × 精度字节数
官方 Discussion #10068 提到没有简单的现成公式，但 server 启动时会打印 allocated GPU mem for context 的摘要。你可以从日志中观察。

针对 5060 Ti 16GB 的实践建议：

  • 7B/8B Q5_K_M + -c 4096：KV cache 约 1–2 GB，总占用约 7–9 GB，非常舒适。
  • 7B/8B Q5_K_M + -c 8192：KV cache 翻倍到约 2–4 GB，总占用约 8–11 GB，仍有余量。
  • 7B/8B Q5_K_M + -c 32768：KV cache 约 8–12 GB，加上权重可能接近或超过 16GB，需要 careful。

因此 -c 不是“越大越好”，而是“你的显存预算允许多大”。

4. Batch：-b 与 -ub

这是最容易混淆的一对参数。官方 README 的定义：

参数全称默认值控制什么
-b--batch-size2048逻辑最大 batch size；影响 prompt prefill 的吞吐量。
-ub--ubatch-size512物理最大 batch size；控制内存/显存峰值。

逻辑 batch (-b) 是什么？

假设你发送了一个 3000 token 的 prompt。llama.cpp 可以一次性处理这些 token（如果 -b ≥ 3000），而不是逐个处理。更大的 -b 意味着更长的 prompt 可以一次性 prefill，prefill 速度更快。

物理 batch (-ub) 是什么？

显存可能不允许一次性处理 3000 token。物理 batch 是实际一次喂给硬件计算的 token 数。如果逻辑 batch 3000 超过物理 batch 512，llama.cpp 会分多次计算（比如分 6 次，每次 512），但在逻辑上仍视为一次 prefill。

一句话区分
-b 决定"我最多想一次处理多少 token"；-ub 决定"硬件一次最多能吃下多少 token"。通常 -ub ≤ -b。

什么时候调它们？

  • 长 prompt prefill 慢：尝试增大 -b（但要确保 -ub 也足够，或显存能承受）。
  • Prefill 阶段 OOM：降低 -ub，让计算切得更细。
  • 通常默认值对 7B/8B 模型足够；遇到性能瓶颈再调。

5. 并发：-np

-np, --parallel

官方定义："number of parallel sequences"。这是 server 独有的参数，决定同时能处理多少个独立请求。

关键认知：每个 slot 有自己的 KV cache。所以：

总 KV cache ≈ 单个 slot 的 KV cache × np

如果你把 -c 设 8192，-np 设 4，那 KV cache 最大可能用到 4 倍于单 slot 的量。这是 server OOM 的常见原因。

5060 Ti 16GB 的实践建议：

  • 先单并发 (-np 1) 跑通所有功能。
  • Agent 需要并发时，逐步增加 -np，同时用 nvidia-smi 观察显存。
  • 如果显存吃紧，优先降 -c（上下文）或 -np（并发），而不是降量化。

6. 采样参数链：从 logits 到 token

模型每步输出的是 logits（每个候选 token 的原始分数）。采样参数链决定"怎么从 logits 变成实际选中的 token"。官方 server README 列出默认采样链：

penalties;dry;top_n_sigma;top_k;typ_p;top_p;min_p;xtc;temperature

它们按顺序执行，像流水线：

1. penalties
presence_penalty / frequency_penalty 对已出现 token 降权，抑制重复。
2. top_k
只保留概率最高的 k 个候选。硬截断，防止采到明显离谱的 token。
3. top_p (nucleus)
从累计概率达到 p 的候选子集中采样。例如 top_p=0.9 表示只在前 90% 概率质量内选。
4. min_p
过滤掉概率 < 最高概率 × min_p 的候选。相对截断。
5. temperature
对 logits 做温度缩放：logits /= temp。temp→0 时最确定，temp 高时分布更平坦随机。

--temp

默认值约 0.8。这是最常见的调参目标：

  • 低 temp (0.1–0.3)：输出更确定、更保守。适合代码、数学、事实问答。
  • 中 temp (0.5–0.8)：平衡。日常对话默认值。
  • 高 temp (1.0+)：更随机、更有创意。适合头脑风暴、写作。

注意：temperature 不改变模型的知识，只改变采样风格。如果你问一个模型不知道的事，低 temp 不会让它突然知道，只会让它更自信地胡说。

--top-p

默认值约 0.9。 nucleus sampling 的核心思想：不要从全部词汇表里采样，只从"核心候选集"里选。

  • --top-p 1.0：不截断，从全部候选采样（配合高 temp 用）。
  • --top-p 0.1：非常严格的截断，只从最顶尖的候选里选，输出更确定。

--top-k

默认值约 40。硬截断：无论概率分布如何，只保留前 k 个候选。防止模型在奇怪的位置采到生僻 token。

--presence-penalty / --frequency-penalty

两者都用于抑制重复，但机制不同：

  • presence_penalty：token 只要出现过一次，就被惩罚。值越大，越倾向于引入新词。
  • frequency_penalty：token 出现次数越多，惩罚越大。更激进地打击重复。

建议从 0 开始调，遇到明显重复再逐步增加（比如 0.1–0.5）。过高会破坏输出连贯性。

7. 完整启动示例（带注释）

llama-server ^
  -m C:\models\model.Q5_K_M.gguf ^    :: 模型文件
  --host 127.0.0.1 ^                  :: 只监听本机
  --port 8080 ^                       :: HTTP 端口
  -ngl all ^                          :: 尽量全放 GPU
  -c 8192 ^                           :: 上下文 8K
  -b 2048 ^                           :: 逻辑 batch 默认
  -ub 512 ^                           :: 物理 batch 默认
  -np 1 ^                             :: 单并发（先跑通）
  --temp 0.7 ^                        :: 温度
  --top-p 0.9 ^                       :: nucleus sampling
  --top-k 40                          :: top-k 截断
