llama.cpp 的最小心智模型

今天只学一件事：当你启动 llama-server -m model.gguf 时，机器里到底有哪几块东西在协作。

与你的任务的关系：你最终要在 Windows + NVIDIA 5060 Ti 16GB 上跑本地模型，并提供 OpenAI API 给 Agent。后续所有安装、量化选择、显存估算、参数调优，都依赖这一课的模型：权重占空间，KV cache 随上下文增长，GPU offload 决定算在哪里发生，采样参数决定 token 如何被选中。

1. 先给 llama.cpp 一个定位

llama.cpp 官方 README 把它描述为一个高性能本地推理项目：支持 GGUF 模型、1.5 到 8 bit 等多种量化、CUDA/Metal/Vulkan 等后端、CPU+GPU 混合推理，并包含 OpenAI-compatible HTTP server。你可以把它理解成：

不是训练框架
主要用于 inference，也就是“拿训练好的模型来生成 token”。
不是模型本身
模型在 .gguf 文件里；llama.cpp 是读取并执行它的引擎。
不是只有 CPU
官方构建文档支持 CUDA；你的 5060 Ti 应优先走 CUDA 后端。
可以当 API 服务
llama-server 提供 OpenAI-compatible chat/completions endpoint。

2. 一次本地推理的五块拼图

先不要背参数。先看数据流：

GGUF 模型文件 ↓ 读取权重、tokenizer、元数据 llama.cpp runtime ↓ 根据 -ngl 决定哪些层放 GPU，哪些留 CPU Prompt prefill ↓ 为已有上下文建立 KV cache Token decode loop ↓ 每次算出 logits，再按 temp/top-p 等采样 输出 token → 拼成回复

拼图 A：GGUF 文件

GGUF 是 llama.cpp 当前主要使用的模型格式。官方 README 说明 llama.cpp 需要 GGUF 格式模型；其他格式通常要转换。GGUF 通常包含权重、tokenizer 信息、模型元数据。你运行时用：

llama-server -m path\to\model.gguf

因此 -m 不是“选择模型名字”，而是“指定实际要映射进内存的模型文件”。

拼图 B：权重与量化

模型权重是固定参数，加载后会占用 RAM/VRAM。量化就是用更少 bit 编码权重。官方 README 明确列出 llama.cpp 支持多种 1.5-bit 到 8-bit 量化；官方 Tensor Encoding Schemes wiki 展示了不同张量编码方案。

量化粗略直觉常见用途
Q4_K_M常用 4-bit K-quant，中等质量/小体积折中。显存有限时的默认候选。
Q5_K_M比 Q4 更大，通常质量更稳。16GB 显存上跑 7B/8B/部分 14B 时常值得考虑。
Q8_0更接近高精度，但体积大。小模型、质量敏感、显存足够时。

今天先记原则：量化主要影响“权重占用”和“质量/速度折中”；但上下文拉长时，另一个大头是 KV cache。

拼图 C：KV cache

Transformer 生成第 1000 个 token 时，不会每步从零重算前 999 个 token 的全部注意力状态，而是缓存先前 token 的 Key/Value。这个缓存就叫 KV cache。

关键：模型文件大小由权重决定；但运行时内存还会随 context size 和并发数增长。也就是：

总显存压力 ≈ 量化后的模型权重 + KV cache + 计算临时缓冲 + server/驱动开销

所以 16GB 显存不是只问“模型文件是不是小于 16GB”。如果你把 -c 设很大，或者 server 并发开很高，KV cache 会把剩余显存吃掉。

拼图 D：GPU offload

llama.cpp 支持 CPU+GPU 混合推理。CUDA 构建见官方 build docs。常见参数 -ngl / --n-gpu-layers 的含义是：把多少层模型计算放到 GPU。

如果显存够，通常希望尽可能多放 GPU；如果显存不够，就减少 -ngl，让部分层回到 CPU，代价是速度下降。

拼图 E：采样

模型每一步输出的是一组 logits：每个候选 token 的分数。采样参数决定“从这些候选里怎么选下一个 token”。

  • --temp：温度。低温更确定，高温更随机。
  • --top-p：只在累计概率达到 p 的候选集合里采样。

这类参数主要影响输出风格，不会让模型“知道更多”。

3. 把参数放回心智模型

你看到的参数它属于哪块拼图第一直觉
-m model.ggufGGUF 文件加载哪个模型。
-ngl 99GPU offload尽量把层放进 GPU；不够显存就降。
-c 8192KV cache / context更长上下文，但更多运行时内存。
-np 2server 并发 / KV cache更多并发 slot，但 KV cache 需求上升。
--temp 0.7采样输出随机性。
--top-p 0.9采样候选 token 截断范围。

4. 你的 5060 Ti 16GB 的初始策略

后续实操时，我们会验证而不是猜。但初始策略可以是：

  1. 先选 7B/8B 级别 instruct GGUF 模型，避免一开始被显存和模型兼容性干扰。
  2. 量化先选 Q5_K_M 或 Q4_K_M：前者更稳，后者更省。
  3. -ngl 先设大，例如 -ngl 99，让 llama.cpp 尽量 offload；如果 OOM 再降。
  4. -c 先从 4096 或 8192 开始，不要一上来追超长上下文。
  5. 先单并发跑通 OpenAI API，再讨论 -np 并发和 Agent 多请求。
