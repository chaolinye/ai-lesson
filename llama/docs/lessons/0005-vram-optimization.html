Flash Attention、KV cache 量化与显存优化

今天的目标：当你的 16GB 显存遇到"模型想要更多"时，你知道按什么顺序、用什么技术来腾出空间。

与你的任务的关系：你在前三课已经能跑起模型、选好量化、理解参数。但总有一天你会遇到——模型 14B 但显存不够、上下文想拉到 32K 但 OOM、Agent 并发请求太多。这节课就是为这些场景准备的"显存工具箱"。

1. 为什么 16GB 显存不是"16GB 可用"

第一课的心智模型更新：

总 VRAM = 量化后权重 + KV cache × 并发数 + 计算缓冲 + 驱动/系统开销

其中：
- 权重：量化后固定大小。选 Q5_K_M 还是 Q4_K_M 的影响。
- KV cache：随上下文和并发增长。是优化的主要目标。
- 计算缓冲：受 -ub、-b 影响。OOM 时可降。
- 驱动/系统：约 0.5–1.5 GB 不可控开销。

所以你在 nvidia-smi 里看到 16GB，实际留给模型的可能只有 14–15 GB。加上 Windows 自身开销，有时更少。

2. Flash Attention — 默认就开了，但你应知道它省什么

官方 PR #15434 已使 Flash Attention 成为默认（新版本 -fa auto 会在支持时自动启用）。

Flash Attention 解决什么问题？

标准注意力计算需要为每个 token pair 计算注意力分数并存储中间矩阵，这需要 O(n²) 显存（n = 上下文长度）。Flash Attention 通过分块（tiling）技术，不把完整的注意力矩阵存在显存中，而是按块计算并逐步归约。

  • 省的是注意力计算的中间结果，不是权重。
  • 效果：上下文越长，节省越显著。8K 以下差异不大；32K+ 时可能是"能跑"和"OOM"的区别。
  • 速度：大多数 GPU 上 FA 不比标准注意力慢，有时还更快（因为减少显存带宽压力）。
  • 确认方法：启动 server 时看日志是否有 "flash attention" 字样。
显式启用：虽然新版本 auto 通常已启用，但如果遇到 OOM 问题，可显式设 -fa on 确认。

3. KV cache 量化 — 省显存的"大杀器"

原理

KV cache 默认用 f16（每个值 2 字节）存储。官方 PR #2969 引入了 K cache 的 q8_0 量化，PR #4312 扩展为支持所有量化格式。

参数 -ctk 控制 K cache 的数据类型，-ctv 控制 V cache。支持：f32, f16, bf16, q8_0, q4_0, q4_1, q5_0, q5_1, iq4_nl。

效果对比（8B 模型）

设置每 token8K 上下文32K 上下文
f16（默认）~0.5 MB~4.0 GB~16.0 GB
-ctk q8_0 -ctv q8_0~0.25 MB~2.0 GB~8.0 GB
-ctk q4_0 -ctv q4_0~0.125 MB~1.0 GB~4.0 GB

建议：

  • 上下文 ≤ 8K：默认 f16 即可。
  • 上下文 8K–32K：-ctk q8_0 -ctv q8_0 是安全的折中。
  • 上下文 > 32K 或并发高：考虑 -ctk q4_0 -ctv q4_0。
质量影响：KV cache 量化对质量的影响通常小于权重量化。社区经验是 q8_0 KV cache 的质量损失几乎不可察觉。

4. 其他显存优化技术

4.1 降低 GPU offload（-ngl）

如果总显存不够，把部分层从 GPU 移回 CPU。代价是速度下降。对于 decoder-only 模型，靠近输出的层（后几层）的每 token 计算量其实不大，留在 CPU 影响相对较小。

4.2 降低 ubatch（-ub）

物理 micro-batch 控制每一步送入 GPU 的数据量。降低 -ub 可以降低 prefill 阶段的峰值显存，但 prefill 速度会变慢。作为 OOM 安全阀很有效。

4.3 Context shift（--context-shift）

当上下文超出限制时，滑动窗口丢弃最早的 token，而不是报错。这在长对话场景中很有用，但会丢失早期上下文。

4.4 模型层面：选更低量化或更小模型

如果优化技术都用完了还是不够，最终的退路是：选更小模型（7B→3B）、更低量化（Q5_K_M→Q4_K_M→Q3_K_M）。

5. 场景决策树

当你遇到 OOM 时，按这个顺序排查：

遇到 OOM？
│
├─ 降低 -ub（安全阀，立即见效）
│
├─ 确认 -fa on（Flash Attention）
│
├─ 确认 -ngl 不是设太高（尤其是大模型）
│
├─ 降低 -c（上下文大小）
│
├─ 降低 -np（并发数）
│
├─ 启用 KV cache 量化：-ctk q8_0 -ctv q8_0
│
├─ 降低量化：Q5_K_M → Q4_K_M → Q3_K_M
│
└─ 换更小模型：14B → 8B → 7B → 3B

6. 实战：不同场景的 16GB 配置

场景 A：8B Q5_K_M，8K 上下文，单并发
-ngl all -c 8192 -fa on -np 1 -ctk f16 -ctv f16
约 11.5 GB，舒适。
场景 B：8B Q5_K_M，32K 上下文，单并发
-ngl all -c 32768 -fa on -np 1 -ctk q8_0 -ctv q8_0
权重 5.5 GB + KV cache ~8 GB = ~15.5 GB（含余量临界）。如果 OOM 试 -ctk q4_0。
场景 C：14B Q4_K_M，8K 上下文，单并发
-ngl all -c 8192 -fa on -np 1 -ctk q8_0 -ctv q8_0
权重 ~8.2 GB + KV cache ~3.5 GB + 余量 ~2 GB ≈ 13.7 GB。可行。
场景 D：8B Q4_K_M，8K 上下文，4 并发（Agent 多请求）
-ngl all -c 8192 -fa on -np 4 -ctk q8_0 -ctv q8_0
权重 ~4.1 GB + KV cache ×4 ≈ 8 GB + 余量 ~2 GB ≈ 14.1 GB。有风险，可再压 -ctk q4_0。

7. 如何验证优化效果

  1. nvidia-smi：观察显存使用量变化。
  2. server 日志：启动时打印 offloaded layers、VRAM 使用。
  3. 实验：改变一个参数，观察显存变化，确认你的理解。
  4. 生成质量：量化 KV cache 后对比输出，确认可接受。
