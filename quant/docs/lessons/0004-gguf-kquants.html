Lesson 4: GGUF & K-Quants

让你的模型在任意硬件上运行——CPU、Mac、低端 GPU

前面三节课讲的量化方法（GPTQ、AWQ）有一个共同的前提：你有一块 NVIDIA GPU。

但现实是：

  • 你可能只有一台 MacBook（Apple Silicon）
  • 你可能只有一台老旧的 CPU-only 服务器
  • 你的 GPU 显存太小（6GB / 8GB），跑 GPTQ 的 inference engine 都装不下
  • 你只是想用 Ollama 拉下来一个模型就跑

在这些场景下，GGUF 就是答案。

📌 最重要的一个区分：GGUF 不是量化算法！
GGUF 是一个文件格式——一个打包了权重、分词器、模型元数据、量化参数的单文件容器。
里面的权重用什么方法量化的，是另一回事（K-Quants、I-Quants、Q8_0……）。

GGUF 的身世

GGUF = GPT-Generated Unified Format，由 Georgi Gerganov（llama.cpp 的作者）开发。它是 GGML 格式的继任者：

GGML (旧) → GGUF (当前) → (未来?)

2023 年 8 月，GGML 被 GGUF 取代，原因是 GGML 的格式设计无法扩展——每增加一个功能就得改二进制格式。GGUF 使用了更灵活的 key-value 元数据系统，可以向前兼容。

⚠️ 不要搞混：很多人说「GGUF 量化」，准确说法应该是「用 GGUF 格式 + K-Quant 量化方案」。GGUF 可以装任何精度的权重——从 F16（未量化）到 Q2_K（极低精度）。

K-Quants：层级量化方案

llama.cpp 里的量化主要有三代：

世代命名规则特点状态
Legacy Q4_0, Q4_1, Q5_0, Q5_1, Q8_0 简单的 block-wise 量化，所有 layer 一视同仁 🟢 仍可用，但不再推荐
K-Quants Q2_K, Q3_K_S/M/L, Q4_K_S/M, Q5_K_S/M, Q6_K 两层级（block + sub-block），不同 layer 分配不同 bit 🟢 当前标准，推荐
I-Quants IQ1_S, IQ2_XS, IQ3_S, IQ4_XS, IQ4_NL Importance-aware，用 Hessian 信息辅助，极低位 🟡 SOTA，但更慢

K-Quant 的命名解码

名字如 Q4_K_M 是有规律的：

部分含义例子

S / M / L大小/质量偏好S=小, M=中, L=大

🏆 黄金法则：如果你不知道选哪个——Q4_K_M。这是 Ollama 的默认选项，llama.cpp 社区公认的「什么都能跑、跑起来都不差」的选项。

K-Quant 的工作原理

K-Quant 的两个核心设计：

1. 层级 Block 结构

不是像 GPTQ/AWQ 那样按 group_size 分块，而是用两层级结构：

权重矩阵的一行 (比如 256 个权重):
┌──────────────────────────────────────────────────────────────┐
│ Block 0 (32 weights) │ Block 1 (32) │ Block 2 (32) │ ...    │
│ scale=0.12, min=-0.8 │ scale=0.15   │ scale=0.09   │        │
│  ┌──────┬──────┬──┐  │              │              │        │
│  │sub-blk│sub-blk│  │  │              │              │        │
│  │scale=0.11 │...│  │  │              │              │        │
│  └──────┴──────┴──┘  │              │              │        │
└──────────────────────────────────────────────────────────────┘

每 32 个权重形成一个 sub-block，几个 sub-block 组成一个 block（通常是 256 个权重）。Sub-block 有自己的 scale（相对于 block scale 编码以节省空间）。

2. 混合精度分配

K-Quant 不要求所有权重层用同一种精度。一个 Q4_K_M 的模型中：

  • Attention 的 Q/K/V/O 层：分配更多 bit（5~6 bit）——它们对输出影响更大
  • FFN 的 gate/up/down 层：分配较少 bit（3~4 bit）——对精度不那么敏感
  • Embedding / LM Head：通常保持 F16 或 Q8

这种混合精度策略使 Q4_K_M 的实际效果接近 Q5，但大小只有 Q4。这就是 K-Quant 的魔术。

💡 与 GPTQ/AWQ 的关键区别：
GPTQ 和 AWQ 是算法——它们决定如何量化。GGUF K-Quant 是方案——它决定量化的格式和分层策略。你不必纠结哪个更好——它们服务不同的硬件生态。

GGUF 的推理生态系统

GGUF 最大的优势是生态——几乎所有本地推理工具都支持它：

工具特点推荐场景
llama.cpp底层 C++ 引擎，CLI技术用户，最大控制权
Ollama一键拉模型，命令行友好✅ 日常使用首选
LM StudioGUI 界面，内置模型浏览器不想碰命令行
KoboldCPP专为角色扮演/写作优化创意写作
text-generation-webui功能全面的 Web UI需要复杂配置时

而且 GGUF 支持层级 GPU 卸载：比如你的 GPU 只有 6GB 显存，可以把一部分 transformer 层放在 GPU 上跑，其余在 CPU 上跑，你的 8GB 显存 + 32GB 内存就能跑 70B 模型。

动手环节：体验 GGUF 的量化矩阵

📝 练习：理解 K-Quant 的层级量化方案

下面的代码模拟 K-Quant 的核心思想——两层级 block + sub-block 量化，并对比与简单 block 量化的效果差异。

import torch
import math

def simple_block_quantize(w, block_size=32, bits=4):
    """简单的 block-wise 量化（类似 Q4_0）"""
    q_max = 2**(bits - 1) - 1
    n = w.numel()
    result = torch.zeros_like(w)
    
    for i in range(0, n, block_size):
        block = w.flatten()[i:i+block_size]
        scale = block.abs().max() / q_max
        q = torch.round(block / scale).clamp(-q_max, q_max)
        result.flatten()[i:i+block_size] = q * scale
    
    return result.reshape(w.shape)

def kquant_simulate(w, bits=4):
    """
    模拟 K-Quant 的两层级结构（教学简化版）
    - 对「重要」的通道分配更多精度
    - sub-block 有自己的相对 scale
    """
    q_max = 2**(bits - 1) - 1
    result = torch.zeros_like(w)
    
    # 模拟：根据权重标准差判断「重要性」
    importance = w.std(dim=0)
    threshold = torch.quantile(importance, 0.75)
    
    for col in range(w.shape[1]):
        col_w = w[:, col]
        is_important = importance[col] >= threshold
        
        if is_important:
            # 重要通道：更小 block + 更多 bits（模拟分配更多精度）
            block_size = 16
            effective_bits = bits + 1  # 相当于更多精度
            eff_q_max = 2**(effective_bits - 1) - 1
        else:
            block_size = 32
            effective_bits = bits
            eff_q_max = q_max
        
        # block-wise 量化
        for i in range(0, len(col_w), block_size):
            block = col_w[i:i+block_size]
            scale = block.abs().max() / eff_q_max
            q = torch.round(block / scale).clamp(-eff_q_max, eff_q_max)
            result[i:i+block_size, col] = q * scale
    
    return result

# 构造模拟数据
torch.manual_seed(42)
# 权重矩阵：有些通道方差大（重要），有些方差小
W = torch.randn(128, 64) * 0.1
# 让一半的通道方差更大
W[:, 32:] *= 3.0

naive_q = simple_block_quantize(W, block_size=32)
kq_sim = kquant_simulate(W)

naive_mse = ((W - naive_q) ** 2).mean().item()
kq_mse = ((W - kq_sim) ** 2).mean().item()

print("=" * 55)
print("简单的 block-wise  vs  K-Quant 风格的层级量化")
print("=" * 55)
print(f"Block-wise 简单量化 MSE:   {naive_mse:.6f}")
print(f"K-Quant 风格层级量化 MSE:  {kq_mse:.6f}")
print(f"改进:                      {(naive_mse - kq_mse) / naive_mse * 100:.1f}%")

# 分别看重要通道和非重要通道
print(f"\n重要通道（高方差）:")
mse_imp_naive = ((W[:, 32:] - naive_q[:, 32:]) ** 2).mean().item()
mse_imp_kq = ((W[:, 32:] - kq_sim[:, 32:]) ** 2).mean().item()
print(f"  Block-wise: {mse_imp_naive:.6f}  K-Quant: {mse_imp_kq:.6f}")

print(f"\n非重要通道（低方差）:")
mse_non_naive = ((W[:, :32] - naive_q[:, :32]) ** 2).mean().item()
mse_non_kq = ((W[:, :32] - kq_sim[:, :32]) ** 2).mean().item()
print(f"  Block-wise: {mse_non_naive:.6f}  K-Quant: {mse_non_kq:.6f}")
试试这些扩展：
• 把重要通道的比例从 50% 改到 10%（更接近真实 LLM）——效果变化大吗？
• 把 block_size 从 32 改到 64 或 128——更大的 block 省存储但损失多少精度？
• 如果你只有 8GB 显存，跑一个 7B 模型应该选 Q4_K_M 还是 Q5_K_M？实际大小差多少？

如何选 GGUF 量化类型？一张决策树

你的显卡/内存能装下多大的模型？
│
├─ < 4GB → Q2_K 或 Q3_K_M（但别期待推理质量好）
│
├─ 4~6GB → Q4_K_S 或 Q4_K_M（优先 Q4_K_M）
│
├─ 6~8GB → Q4_K_M（完美适配 7B 模型）
│
├─ 8~12GB → Q5_K_M（升级精度）
│
├─ 12~24GB → Q6_K 或 Q8_0（几乎无损）
│
└─ > 24GB → 直接跑 F16/BF16，或者跑更大的模型！
