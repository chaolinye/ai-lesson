Windows CUDA 版 llama.cpp 与第一个 API 服务

今天的胜利条件：你能解释“为什么要 CUDA 版”，并知道两条安装路线；然后用 llama-server 启动一个 OpenAI-compatible 本地服务。

与你的任务的关系：你的最终目标是让 Agent 调本地模型。Agent 不关心你模型是怎么跑的，它只需要一个 OpenAI-compatible endpoint；但你必须关心 CUDA、GGUF、显存和上下文，因为这些决定服务是否快、稳、可用。

1. 先理解：安装 llama.cpp 其实是在选择“执行后端”

llama.cpp 是推理引擎；同一个 .gguf 模型可以在 CPU、CUDA、Vulkan 等后端上执行。官方 README/构建文档列出 CUDA 后端用于 NVIDIA GPU。你的 5060 Ti 16GB 是 NVIDIA GPU，所以学习路线应默认追求 CUDA 加速。

模型文件 model.gguf │ ▼ llama.cpp 可执行文件：llama-server.exe │ ├─ CPU backend：能跑，但通常慢 └─ CUDA backend：把大部分矩阵计算交给 NVIDIA GPU

所以“装好了 llama.cpp”不够精确。真正的问题是：

  • 我是否有 llama-server.exe？
  • 它是否带 CUDA 支持？
  • 启动时模型层是否真的 offload 到 GPU？

2. 两条安装路线：先快跑，再可控

路线 A：Winget / Releases 预构建

适合先快速跑通 server。官方 install 文档给出 Windows 上可用：

winget install llama.cpp

优点：简单。风险：你需要确认安装出来的二进制是否符合你的 CUDA 需求。

路线 B：源码 CUDA 构建

适合你这种会 C/C++、想理解原理和参数的人。官方 build 文档的 CUDA 构建核心是：

cmake -B build -DGGML_CUDA=ON
cmake --build build --config Release

优点：可控。代价：要处理 VS/CMake/CUDA Toolkit 环境。

建议：如果你现在就要跑，先试路线 A；如果发现 GPU 没用上，或后续要稳定复现环境，再走路线 B。学习上我们两条都理解，但实操优先用能最快反馈的路线。

3. 路线 A：预构建安装检查

在 PowerShell 中执行：

winget install llama.cpp
llama-server --help

你要观察的不是把 help 全背下来，而是确认命令存在，并在参数里看到类似：

  • -m, --model
  • -c, --ctx-size
  • -ngl, --gpu-layers, --n-gpu-layers
  • --host / --port

官方 server README 说明 -ngl 是 “max. number of layers to store in VRAM”，新版本默认可能是 auto。这句话很重要：VRAM 里放的是模型层，不是“整个程序”。

4. 路线 B：源码 CUDA 构建的最小闭环

如果你要自己构建，先准备：

  • Visual Studio 2022，包含 “Desktop development with C++”。官方 build docs 特别提醒 Windows 构建要用 VS Developer PowerShell/Command Prompt。
  • Git。
  • CMake。
  • NVIDIA CUDA Toolkit。

然后在 Developer PowerShell for VS 2022 中：

git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp
cmake -B build -DGGML_CUDA=ON
cmake --build build --config Release -j

构建完成后，常见可执行文件会在类似路径：

build\bin\Release\llama-server.exe

如果遇到 No CUDA toolset found：这通常不是 llama.cpp 参数问题，而是 CMake 没找到 CUDA Toolkit 或 VS/CUDA 集成环境。先检查 nvcc --version、CUDA Toolkit 是否安装、是否在 VS Developer shell 中运行。

5. 选择第一个模型：不要一开始挑战极限

你的显卡 16GB 很适合 7B/8B instruct 模型。第一轮目标是 API 跑通，不是压榨最大模型。

选择理由
7B/8B instruct GGUF足够小，容易验证 CUDA/offload/API 全链路。
Q5_K_M质量和体积折中，16GB 上通常比 Q4 更安心。
Q4_K_M如果想留更多显存给上下文/KV cache，选它。
暂不选 30B+容易让你把“模型太大”的问题误判成“安装失败”。

从 Hugging Face 找模型时，关键词通常是：模型名 + GGUF + Instruct。后续我们会单独上一课讲如何读模型卡和选择量化。

6. 启动第一个 OpenAI-compatible server

假设你下载了：

C:\models\your-model.Q5_K_M.gguf

启动：

llama-server ^
  -m C:\models\your-model.Q5_K_M.gguf ^
  --host 127.0.0.1 ^
  --port 8080 ^
  -ngl auto ^
  -c 4096

如果你的版本不接受 -ngl auto，试：

-ngl 99

含义不是“真的有 99 层都存在”，而是常用作“尽量多 offload”。新版本 help 若写支持 auto/all，优先用它们。

7. 测试 API

PowerShell 里可以先用 curl 测试。Windows 的 curl 引号容易烦，下面给一个单行版本：

curl http://127.0.0.1:8080/v1/chat/completions -H "Content-Type: application/json" -d "{\"model\":\"local\",\"messages\":[{\"role\":\"user\",\"content\":\"用一句中文介绍 llama.cpp\"}],\"temperature\":0.7}"

如果返回 JSON，并且里面有 assistant 内容，你已经完成 Agent 所需 API 的最小闭环。

8. 如何判断 CUDA 真在工作？

  1. 看 server 启动日志。通常会打印 backend、offloaded layers、VRAM 使用等信息。
  2. 开另一个终端运行：
    nvidia-smi
    观察显存占用和 GPU 利用率。
  3. 如果 -ngl 从低到高时速度和显存明显变化，说明 offload 起作用。

9. 小练习：把命令拆成概念

Q1. --host 127.0.0.1 和 --port 8080 解决的是模型质量问题、网络入口问题，还是显存问题？

Q2. 为什么第一轮建议 -c 4096，而不是直接开到 32768？

Q3. 如果 server 能启动但生成很慢，你首先怀疑 CPU/GPU offload、temperature，还是 top-p？
